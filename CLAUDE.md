# Working agreement

This repo is delivered with the `sfmate` + `mate-core` plugins (four agents across them; the operator orchestrates; CI is the only mechanical gate).

0. **Proportionality:** the request's weight sets the process weight. A small, unambiguous change — a field, a label, a config value, a one-file fix — skips refinement entirely: no analyst, no work-item review, and never a split into multiple tickets. Go straight to Build under whichever is lightest — the existing ticket that covers it, one plainly-written task filed without ceremony, or a ticketless chore branch where the repo's conventions allow one. Review and Ship apply unchanged — the diff gate never scales away. If the request is ambiguous, one clarifying question beats a drafted story; when genuinely unsure which path, ask — never default to the full loop for something the operator said in a sentence.
1. **Triage** (when a bug report arrives): produce ONE verdict before anyone spends build time — confirmed defect (root-cause analysis on the issue, plus a linked fix item), duplicate, not-a-bug, or insufficient-info naming exactly what is missing. A confirmed fix item flows through the loop below like any other work, at the weight step 0 assigns it.
2. **Refine:** for work that needs designing, `mate-core:analyst` drafts a plain story or task — problem, 5–7 acceptance criteria as **rule + example** pairs, out-of-scope, and a `Build:` line (`standard` or `red-first — <why>`, judged from the code it grounded in). Before the operator sees it, `mate-core:reviewer` (work-item mode) attacks the draft — an untestable or existence-only criterion is CRITICAL, an under-called `Build:` line is MAJOR — and findings go back to the analyst first. The operator edits or approves the reviewed body by reading it. No labels, no stages.
3. **Build:** one git worktree + branch per issue, cut from `origin/main`. Spawn `sfmate:developer` with the issue, the working directory, and an org alias (scratch org per worktree — see the plugin's `org` skill; `lab/org-setup.sh` creates one). The issue's `Build:` line sets its discipline — red-first means failing tests before implementation; the developer may escalate itself to red-first when the code reveals unnamed risk, never downgrade. It implements, tests for real, and reports honestly — including what it could not verify.
4. **Review:** spawn `mate-core:reviewer` (read-only) on the committed diff with the work item, the developer's report, and the path to sfmate's `rules/` directory. Critical and major findings are fixed and the **delta** re-reviewed; **done is a post-fix re-review with zero critical / zero major — minors never block, and a fix agent's "fixed" claim never substitutes for the re-review.** The loop runs without the operator; consult them only for a contested finding, a fix that would change scope, or a loop not converging after two fix rounds.
5. **Ship:** push, open the PR, CI green on the pushed SHA, the operator merges.
6. **Always:** verification is real commands and real exit codes, never an agent's claim. **The session keeps the `NOTES.md` ledger itself:** when it observes a hurt — the operator correcting the same thing twice, a red CI a local check should have caught, a stalled loop, redone work, plain frustration — it appends the one-line entry, says so, and first checks the existing lines: a match means the rule of two fired, and the session proposes the smallest mechanism that closes it — applied here when the fix is local, or filed as an issue on the plugin's repo (publicanus/sfmate or publicanus/mate-core) with both ledger lines as evidence when the fix is a framework change. The operator can always dictate a line; they should never have to remember to. An edit to this working agreement itself is always a framework change: apply it here AND mirror it to the canonical agreement in sfmate's GUIDE (a PR on publicanus/sfmate) in the same session — a consumer copy that silently diverges in substance is itself a hurt.

## Repo facts

- Tracker: GitHub Issues (`gh`).
- Salesforce DX project (dreamhouse fork). Org strategy: scratch orgs off the DevHub; `lab/org-setup.sh <alias>` provisions one.
- CI: `.github/workflows/ci.yml` (prettier, lint, LWC unit tests). Org-based validation per the plugin's `cicd` skill templates is a candidate once DevHub secrets are configured — see the plugin README.
- `npm run test:unit` for LWC; Apex tests run against the org via `sf apex run test` (always `--wait`).
